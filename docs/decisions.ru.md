# Ключевые решения (в формате ADR)

Сжатые записи решений, сформировавших систему — только контекст и
последствия, без клиентских данных и кода.

## ADR-1: Git как источник истины
**Контекст.** Система определяется множеством декларативных входов
(таксономии ниш, каталоги клиентов, гео-конфиги, шаблоны).
**Решение.** Хранить их все файлами в git; БД — производная runtime-копия.
**Последствия.** Бесплатный audit-трейл (git log), тривиальный откат,
рутинные правки без БД, «клон = развёртывание». Цена — шаг `sync` для
загрузки БД перед обслуживанием.

## ADR-2: Детерминированный конвейер, LLM только по краям
**Контекст.** Структура сайта должна быть воспроизводимой и проверяемой;
вывод LLM — нет.
**Решение.** Конвейер (таксономия → чертёж → ТЗ страниц) чист и
детерминирован; LLM ограничен черновиками контента за адаптером.
**Последствия.** Прогоны диффаются и тестируются; смена провайдера
локальна; в большей части системы нет недетерминизма для отладки.

## ADR-3: Пофайловый source-tracking
**Контекст.** Регенерация не должна уничтожать ручные правки, но
авто-поля должны обновляться.
**Решение.** Каждое мульти-источниковое поле хранит `source`
(`yaml`/`csv`/`llm`/`manual`/`api`); `sync` пропускает не-`yaml` поля и
обновляет остальные — на поле, не на строку. «Reset to auto» снова
открывает поле.
**Последствия.** Люди и конвейер безопасно правят одни и те же строки;
это и есть фича, делающая масштабную генерацию управляемой.

## ADR-4: Серверный рендеринг вместо SPA
**Контекст.** Поиск смещается к AI-движкам, читающим HTML без выполнения
JavaScript.
**Решение.** Рендерить каждую страницу на сервере (Jinja2 + HTMX),
отдавать готовый HTML со Schema.org JSON-LD; без клиентской загрузки
контента.
**Последствия.** Сильная позиция и в классическом SEO, и в GEO, низкий
TTFB, один стек. Интерактивность — из серверных партиалов, не из SPA.

## ADR-5: Runtime-копия в PostgreSQL, отдельно от git-файлов
**Контекст.** Парсить десятки yaml на каждый HTTP-запрос слишком
медленно, а редакторам не нужно править yaml.
**Решение.** `sync` грузит файлы в PostgreSQL; приложение в рантайме
читает только БД.
**Последствия.** Быстрое чтение, поверхность ручного редактирования,
история изменений, мультитенантность и аналитика — ценой шага sync и
контракта файлы↔БД.

## ADR-6: Сначала мультитенантность на одной БД
**Контекст.** Много клиентов/филиалов, но большинство небольшие.
**Решение.** Одна БД со скоупингом по `tenant_id` через middleware +
ORM-фильтр-событие; изолировать тенанта позже через ETL в выделенный
инстанс.
**Последствия.** Дёшево стартовать; код всегда фильтрует, поэтому
эскалация — перемещение данных, а не переписывание.

## ADR-7: SQLAdmin + RBAC вместо своей админки
**Контекст.** Нетехническим редакторам нужны CRUD, превью и публикация.
**Решение.** Использовать SQLAdmin поверх ORM-моделей с ролями и
пер-строковым tenant-скоупом, а не строить админку с нуля.
**Последствия.** Большая часть бэк-офиса бесплатно; кастомные actions
(re-render, генерация) добавляются по мере нужды; меньше своего кода.

## ADR-8: Атомарный деплой с откатом по health-check
**Контекст.** Плохая публикация не должна оставлять живой сайт сломанным.
**Решение.** Сборка в версионированную папку, переключение заменой
симлинка, health-check ключевых URL, автоматический откат при не-200;
хранить последние N сборок.
**Последствия.** Безопасные обратимые публикации; откат из истории в
один клик.

## ADR-9: Свой FileLock для shared mutable файлов
**Контекст.** Параллельные задачи клиентов одной ниши read-merge-write
один общий семантический банк, теряя обновления друг друга.
**Решение.** Кроссплатформенный advisory FileLock (sidecar-lockfile, pid
+ таймстемп, stale-reclaim) сериализует load-merge-write; атомарные
записи + retry дополняют картину.
**Последствия.** Нет потерянных обновлений на общих файлах; понятная
ошибка таймаута вместо тихой порчи.

## ADR-10: Декаплинг семантики
**Контекст.** Сбор семантики (спрос, дедуп, таксономии) — отдельная
забота, которая хочет расти независимо (мультиниша, векторное ядро).
**Решение.** Вынести её в отдельную платформу; этот генератор потребляет
готовое ядро через тонкий шов `SemanticSource` (cache-by-default, HTTP
когда апстрим доступен, фолбэк на кэш).
**Последствия.** Генератор больше не собирает семантику — он компилирует
из готового ядра; две системы развиваются независимо.

## ADR-11: Caddy как reverse proxy
**Контекст.** Десяткам доменов нужны TLS и роутинг по Host с минимумом
эксплуатации.
**Решение.** Один инстанс Caddy роутит по Host, выпускает автоматические
Let's Encrypt сертификаты на каждый домен и обрабатывает
редиректы/сжатие.
**Последствия.** Пер-доменный HTTPS почти без конфига; прокси — ещё и
точка переключения для будущей изоляции тенантов.

## ADR-12: SQLite для dev/тестов, паритет PostgreSQL в CI
**Контекст.** Тесты должны быть быстрыми локально, но прод — на PostgreSQL.
**Решение.** Гонять сюиту на in-process SQLite для скорости и ту же
сюиту на реальном сервисе PostgreSQL в CI, чтобы ловить расхождения
DDL/поведения.
**Последствия.** Быстрая локальная итерация без потери прод-точности.
