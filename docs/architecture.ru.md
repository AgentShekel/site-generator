# Архитектура

Публичное архитектурное описание — только принципы и структура. Без
клиентских данных, исходного кода и проприетарных деталей.

## Слои

1. **Git — источник истины.** Всё, что определяет систему, лежит файлом
   в репозитории: скрипты конвейера, DNA ниши (таксономии), мастер-
   таксономии клиентов, гео-конфиги филиалов, шаблоны, миграции БД.
   Зачем: git log — встроенный audit-трейл; откат бесплатен; рутинные
   правки не требуют БД; клон репозитория разворачивает всю систему.

2. **Конвейер — детерминированные скрипты.** Идемпотентный Python
   превращает источники в промежуточные артефакты: `mapper` (таксономия
   + DNA → чертёж сайта), `generator` (чертёж → ТЗ страниц),
   `tech_deploy` (sitemap.xml + robots.txt). Повторный прогон даёт
   идентичный результат.

3. **БД — runtime-копия.** Шаг `sync` загружает yaml + артефакты в
   PostgreSQL; дальше CMS читает только из БД (парсить yaml на каждый
   HTTP-запрос — медленно). БД даёт быстрое чтение под SSR, ручные
   правки, историю изменений, мультитенантность и аналитику.

4. **Админка — SQLAdmin + RBAC.** Web-UI поверх ORM-моделей: список /
   форма / фильтры на каждую модель, кастомные actions (re-render,
   запуск генерации), авторизация с ролями (owner / admin / editor /
   viewer) и пер-строковым tenant-скоупом.

5. **Фронтенд — серверный рендеринг (Jinja2 + HTMX).** Каждая страница
   рендерится на сервере; браузер получает готовый HTML без JS-загрузки
   контента. Критично для GEO: AI-краулеры парсят HTML без выполнения
   JS. Страница = запись в БД + упорядоченный список типизированных
   блоков + meta (title, description, Schema.org JSON-LD). Брендовые
   поверхности берут бренд клиента; гео-поверхности — филиал; без
   хардкода клиента.

6. **Reverse proxy — Caddy.** Один инстанс роутит каждый домен на бэкенд
   по Host-заголовку, терминирует автоматический Let's Encrypt SSL на
   каждый домен и обрабатывает HTTP→HTTPS, сжатие и кэш статики.

## Поток данных (заведение → онлайн)

```
входные данные (каталог, гео) + DNA ниши
        |  детерминированный конвейер
        v
ТЗ страниц + sitemap + robots
        |  sync
        v
PostgreSQL (контент + source на поле)
        |--> редактор (админка: правка / превью / публикация)
        `--> деплой -> версионированная сборка -> Caddy -> посетитель / AI-краулер
```

## Схема БД (эскиз)

Все таблицы несут `tenant_id` + таймстемпы (ниже опущены).

- **clients** — slug, name, brand_name, niche, plan, active
- **branches** — client_id, slug, name, domain (unique), city, district,
  address, metros, og_image_url
- **services** — branch_id, category, slug, name, price, source, order
- **pages** — branch_id, slug, template_name, title, description, h1,
  canonical, meta_json, schema_org_json_ld, published, source, locked
- **blocks** — page_id, block_type, order, content_json, source
- **lsi_phrases** — page_id, phrase, volume, source
- **users** — email, role, tenant_scope, password_hash
- **audit_log** — user_id, entity, action, diff_json, at
- **deploy_history** — client_id, branch_id, version, status, log
- **jobs** — type, payload, status (async-работа: генерация контента и т.п.)

## Source tracking

Каждое поле, которое может прийти из разных источников, хранит `source`
(`yaml` / `csv` / `llm` / `manual` / `api`). При `sync`: запись с
source ≠ `yaml` пропускается; новые или `yaml`-записи обновляются — на
поле, а не на строку. «Reset to auto» возвращает полю `yaml`, и
следующий sync его обновит. Именно это даёт регенерации и ручному
редактированию безопасно сосуществовать.

## Мультитенантность

На старте — одна БД; каждый запрос автоматически скоупится
`WHERE tenant_id = ?` через middleware + ORM-фильтр-событие. Когда
клиент перерастает общий ресурс: поднимается отдельный compose-инстанс,
строки этого тенанта копируются ETL, его домены переключаются на новый
инстанс через прокси верхнего уровня, старая копия удаляется. Код всегда
фильтрует, поэтому эскалация — это ETL, а не переписывание.

## Жизненный цикл деплоя

1. **Pre-flight validate** — обязательные поля на месте, нет битых
   ссылок, sitemap валиден.
2. **Build** — рендер всех страниц тенанта в версионированную папку.
3. **Atomic switch** — прокси переключается на новую папку заменой
   симлинка.
4. **Health check** — запрос `/`, `/sitemap.xml`, `/robots.txt` +
   несколько случайных страниц; не-200 откатывает на предыдущую сборку.
5. **Record** — запись в `deploy_history`. Последние N сборок хранятся
   для отката в один клик.

## Надёжность

- **FileLock** — кроссплатформенный advisory-lock (sidecar-lockfile с
  pid + таймстемпом, stale-reclaim когда владелец мёртв) сериализует
  load-merge-write по общим семантическим банкам, предотвращая
  потерянные обновления при параллельных задачах над одним файлом.
- **Атомарные записи** — каждая запись это tmpfile + `os.replace`.
- **Retry** — транзиентные сбои ретраятся вокруг лока.

## SEO-эндпоинты

Каждый опубликованный домен отдаёт: SSR home + страницы услуг /
категорий / статей, `/sitemap.xml`, forward-совместимый
`/sitemap-index.xml`, `/robots.txt` (со строками Sitemap) и брендовый
404 (HTML или JSON). Smoke-шаг в CI прогоняет эти контракты через
in-process ASGI на каждый push и падает до merge, если контракт сломан.

## Тестирование

Пирамида тестов: unit (функции конвейера, модели), integration (API
через in-process ASGI + SQLite, плюс та же сюита на реальном PostgreSQL
для прод-паритета), smoke-эндпоинты (SEO-контракты) и post-deploy
health-чеки. CI гоняет pytest + полный конвейер + SEO-гарды (нет дублей
H1, sitemap=страницы, контент-фильтры) на каждый push.

## Чего намеренно НЕТ в архитектуре

SPA-фреймворков (бьют по TTFB и GEO), headless CMS (лишний стек),
микросервисов (избыточно на этом масштабе), своей ORM / очереди / RBAC
(SQLAlchemy / arq / SQLAdmin уже подходят). Система остаётся лёгким
монолитом с понятным путём масштабирования только там, где нужно.
